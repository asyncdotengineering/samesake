You've wired search. You've seen `pushDocuments`, `enrich`, and `index` do their thing — maybe from a script, maybe from the [marketplace guide](/guides/marketplace-search/). The catalog stays fresh because the loop is incremental: unchanged rows are no-ops, changed rows get re-read and re-embedded.

That works until it doesn't. Your process restarts during deploy. A model call times out halfway through `index`. A webhook fires at the same moment the cron runs. Someone SSHs in and kills the Node process. Without durability, you're guessing what finished and what didn't.

This page is about **where the loop runs** and **what catches it when something breaks**.

## The loop, as five steps

The continuous pipeline is really five matcher calls. Three run often; two run on a slower cadence:

| Step | Matcher call | Cadence | What it does |
|------|--------------|---------|--------------|
| 1 | `pushDocuments(project, collection, rows)` | Every few minutes + on webhooks | Upsert raw listings; fingerprint changes reset enrich/index |
| 2 | `enrich(project, collection)` | After push | Model reads dirty rows; gate quarantines bad ones |
| 3 | `index(project, collection)` | After enrich | Embeds and writes vectors for stale rows |
| 4 | `revalidateImages(project, collection)` | Hourly | Catches same-URL image swaps |
| 5 | `retryFailed(project, collection)` | Hourly | Retries `failed` rows with backoff |

Because each step is **idempotent and incremental**, you don't need clever fan-out. samesake batches internally — your job platform's job is **durability, retries, scheduling, and observability**, not splitting 50,000 products into 50,000 queue messages.

<Aside type="note">
Every row's honest state — `pending`, `ready`, `quarantined`, `failed`, `dead` — lives in the database. See [Pipeline lifecycle](/guides/pipeline-lifecycle/) for the full state machine.
</Aside>

## Why not just `setInterval`?

A timer in a long-running Node process is fine for local dev. For production it falls apart quickly:

- **No retries.** If `index` throws, the timer moves on. Those rows sit `failed` until something else notices.
- **Lost work on restart.** Deploy, crash, scale-to-zero — whatever was in flight is gone.
- **Overlapping runs.** Two crons fire at once and you double your model bill for no reason.
- **No observability.** When search looks stale at 3am, you're grep-ing logs.

A queue or workflow platform gives you **durable steps** (a failed `index` retries without re-running `enrich`), **concurrency guards**, and a dashboard that shows what ran and what didn't.

## The pattern every platform guide uses

Each target guide below shows the same spine:

1. **Scheduled full-loop** — every few minutes: push → enrich → index. Hourly: revalidateImages → retryFailed. This is your safety net even when webhooks work.
2. **Webhook fast-path** (optional) — one product changed → `pushDocuments` with just that row. The next scheduled enrich/index picks it up, or you run them inline in the same workflow.

Keep the three main stages as **separate durable steps**. That's how you get per-stage retries, per-stage timing in your observability tool, and the guarantee that a flaky embed call doesn't force you to re-fetch your whole catalog.

```ts title="matcher.ts — shared setup"
import { createMatcher, llmRerank } from "@samesake/server";
import { products } from "./catalog.ts";
import { geminiEmbed, geminiGenerate } from "./gemini.ts";

export const matcher = createMatcher({
  databaseUrl: process.env.SAMESAKE_DATABASE_URL!,
  apiKey: process.env.SAMESAKE_API_KEY!,
  embed: geminiEmbed,                       // gemini-embedding-2
  generate: geminiGenerate,                 // gemini-3.1-flash-lite
  rerank: llmRerank(geminiGenerate),
});

await matcher.apply("market", { entities: [], collections: [products] });
```

| Guide | Reach for it when… |

| [In-memory](/guides/pipeline-in-memory/) | Local dev or a single long-running process — zero extra infra |

## Did the enrichment actually work?

Enrichment quality is the make-or-break for search — so measure it, don't assume it. After a run,
`matcher.evaluateEnrichment(project, collection, { gold, attributes: fashion.evalAttributes() })`
scores the pipeline's extracted attributes against a human-labeled gold set with **per-attribute
precision / recall / F1** (and flags where the model hallucinated or missed a value). Gate your
enrich-prompt, taxonomy, or confidence-floor changes on it. See
[Measure enrichment accuracy](/guides/eval-enrichment/).
