[Cloudflare Workflows](https://developers.cloudflare.com/workflows/) gives you durable steps on Workers via `step.do`. Cron schedules on the workflow binding auto-create instances — no separate cron Worker. Failed steps retry with configurable backoff.

See [Running the enrich pipeline durably](/guides/enrich-pipeline/) for the five-step model.

## Matcher setup

Your matcher runs inside the Worker (or calls out to a Node service that holds it — same matcher calls either way). For a Worker-hosted matcher, keep the database URL and API key in secrets:

```ts title="src/matcher.ts"
import { createMatcher, llmRerank } from "@samesake/server";
import { products } from "./catalog.ts";
import { geminiEmbed, geminiGenerate } from "./gemini.ts";

export function createCatalogMatcher(env: Env) {
  return createMatcher({
    databaseUrl: env.SAMESAKE_DATABASE_URL,
    apiKey: env.API_KEY,
    embed: geminiEmbed,
    generate: geminiGenerate,
    rerank: llmRerank(geminiGenerate),
  });
}
```

## Workflow class

```ts title="src/workflows/catalog-sync.ts"
import {
  WorkflowEntrypoint,
  type WorkflowEvent,
  type WorkflowStep,
} from "cloudflare:workers";
import { createCatalogMatcher } from "../matcher.ts";
import { fetchChangedRows } from "../catalog-source.ts";

type Env = {
  SAMESAKE_DATABASE_URL: string;
  API_KEY: string;
};

const PROJECT = "market";
const COLLECTION = "products";

export class CatalogSync extends WorkflowEntrypoint<Env> {
  async run(event: WorkflowEvent<unknown>, step: WorkflowStep) {
    const matcher = createCatalogMatcher(this.env);

    await step.do("push", async () => {
      const rows = await fetchChangedRows();
      await matcher.pushDocuments(PROJECT, COLLECTION, rows);
    });

    await step.do("enrich", () => matcher.enrich(PROJECT, COLLECTION));

    await step.do(
      "index",
      {
        retries: { limit: 5, delay: "5 seconds", backoff: "exponential" },
      },
      () => matcher.index(PROJECT, COLLECTION),
    );
  }
}

export class CatalogMaintenance extends WorkflowEntrypoint<Env> {
  async run(_event: WorkflowEvent<unknown>, step: WorkflowStep) {
    const matcher = createCatalogMatcher(this.env);

    await step.do("revalidate-images", () =>
      matcher.revalidateImages(PROJECT, COLLECTION),
    );
    await step.do("retry-failed", () =>
      matcher.retryFailed(PROJECT, COLLECTION),
    );
  }
}
```

## wrangler.toml — bindings and schedules

The `schedules` array on each workflow binding runs the loop on a cron — Cloudflare creates a new instance automatically.

```toml title="wrangler.toml"
name = "samesake-pipeline"
main = "src/index.ts"
compatibility_date = "2026-06-09"

[[workflows]]
name = "catalog-sync"
binding = "CATALOG_SYNC"
class_name = "CatalogSync"
schedules = ["*/5 * * * *"]

[[workflows]]
name = "catalog-maintenance"
binding = "CATALOG_MAINTENANCE"
class_name = "CatalogMaintenance"
schedules = ["0 * * * *"]
```

## Webhook fast-path — create an instance from a Worker

When a product changes, create a workflow instance with that row in `params`, or push directly and let the scheduled sync pick it up:

```ts title="src/index.ts"
import { createCatalogMatcher } from "./matcher.ts";
import { CatalogSync } from "./workflows/catalog-sync.ts";

export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    if (request.method === "POST" && new URL(request.url).pathname === "/webhooks/product") {
      const product = await request.json();

      // Option A: push inline, scheduled sync handles enrich/index
      const matcher = createCatalogMatcher(env);
      await matcher.pushDocuments("market", "products", [product]);

      // Option B: trigger a sync workflow now
      const instance = await env.CATALOG_SYNC.create({
        params: { trigger: "webhook" },
      });

      return Response.json({ ok: true, instanceId: instance.id });
    }

    return new Response("Not found", { status: 404 });
  },
};
```

Use `step.sleep("name", "20 seconds")` between steps if you need deliberate delays — instances in a waiting state don't count toward concurrency limits.

<Aside type="note">
Per-step retry config (`retries: { limit, delay, backoff }`) applies to that `step.do` only. A failure in `index` retries `index` — not `push` or `enrich`.
</Aside>

## Why this fits samesake

Workers + Workflows keep everything at the edge if your app already lives there. samesake's incrementality means each scheduled instance still runs three matcher calls over the whole collection — cheap when only dozens of rows changed. You don't fan out per product; Cloudflare's durability covers per-step retries.
