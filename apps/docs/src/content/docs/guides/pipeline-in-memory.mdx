The simplest way to keep a catalog fresh is a timer in a long-running Node process. No queue, no workflow SDK, no extra accounts. You already have the matcher; you just call it on a schedule.

This is the **zero-dependency** option. It's perfect for local dev and fine for a single server you control. It's not durable across restarts, and it doesn't survive horizontal scale. When you're ready for production, graduate to one of the [durable platform guides](/guides/enrich-pipeline/).

## Shared matcher setup

```ts title="matcher.ts"
import { createMatcher, fashionRerank } from "@samesake/server";
import { products } from "./catalog.ts";
import { geminiEmbed, geminiGenerate } from "./gemini.ts";

export const matcher = createMatcher({
  databaseUrl: process.env.DATABASE_URL!,
  apiKey: process.env.API_KEY!,
  embed: geminiEmbed,
  generate: geminiGenerate,
  rerank: fashionRerank(geminiGenerate),
});

await matcher.apply("market", { entities: [], collections: [products] });
```

## The sync loop with overlap protection

Two things matter: **don't let runs overlap**, and **run maintenance on a slower cadence**.

```ts title="sync.ts"
import { matcher } from "./matcher.ts";
import { fetchChangedRows } from "./catalog-source.ts";

const PROJECT = "market";
const COLLECTION = "products";

let syncRunning = false;

async function syncCatalog() {
  if (syncRunning) {
    console.log("[sync] skipped — previous run still in progress");
    return;
  }
  syncRunning = true;
  try {
    const rows = await fetchChangedRows();
    await matcher.pushDocuments(PROJECT, COLLECTION, rows);
    await matcher.enrich(PROJECT, COLLECTION);
    await matcher.index(PROJECT, COLLECTION);
    console.log("[sync] done");
  } catch (err) {
    console.error("[sync] failed:", err);
  } finally {
    syncRunning = false;
  }
}

async function runMaintenance() {
  try {
    await matcher.revalidateImages(PROJECT, COLLECTION);
    await matcher.retryFailed(PROJECT, COLLECTION);
    console.log("[maintenance] done");
  } catch (err) {
    console.error("[maintenance] failed:", err);
  }
}

// Full loop every 5 minutes
setInterval(syncCatalog, 5 * 60 * 1000);

// Image revalidation + retries every hour
setInterval(runMaintenance, 60 * 60 * 1000);

// Run once on startup so you're not waiting for the first tick
syncCatalog();
```

The guard (`syncRunning`) stops a slow enrich from stacking another full loop on top. Maintenance doesn't need the same guard — it's cheap relative to enrich, and the matcher calls are idempotent.

<Aside type="note">
If `syncCatalog` throws mid-loop, the guard still clears in `finally`, so the next tick retries. But there's **no per-step durability**: a failure during `index` won't resume from `index` — it'll re-run push and enrich too. That's harmless (idempotent) but wasteful. Durable platforms fix exactly this.
</Aside>

## Webhook fast-path (optional)

When a product changes in your store, push just that row. The next scheduled enrich/index picks it up:

```ts title="webhook.ts"
import { matcher } from "./matcher.ts";

export async function onProductUpdated(product) {
  await matcher.pushDocuments("market", "products", [product]);
  // enrich + index happen on the next cron tick — or call them inline if you need instant search
}
```

For near-instant updates, call enrich and index right after push in the webhook handler. Because the loop is incremental, doing both cron *and* inline webhook syncs can't corrupt anything — worst case you do a little extra work.

## What you're trading away

| | In-memory timer | Durable queue/workflow |
|--|-----------------|------------------------|
| Survives process restart | No | Yes |
| Per-step retry (`index` fails, `enrich` doesn't re-run) | No | Yes |
| Horizontal scale (multiple processes) | No — double runs | Yes — concurrency limits |
| Observability dashboard | Your logs | Platform UI |
| Extra infra | None | One service |

<Aside type="tip">
Start here. Ship search. When deploys start killing mid-sync runs or you need a clear audit trail, move to [Inngest](/guides/pipeline-inngest/), [Upstash](/guides/pipeline-upstash/), [Cloudflare Workflows](/guides/pipeline-cloudflare-workflows/), or [Vercel Workflows](/guides/pipeline-vercel-workflows/). The matcher calls don't change — only the wrapper does.
</Aside>
