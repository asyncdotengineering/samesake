# Fashion Search Proof

This proof page translates the parity benchmark in [`examples/fashion-search/PARITY.md`](../examples/fashion-search/PARITY.md) into developer-facing evidence.

It does not claim production dominance. It shows that Samesake can reproduce the fashion-search spike's intent-search quality within tolerance on the same catalog and harness, while making the retrieval stack configurable in TypeScript.

## Corpus

The benchmark used:

- 4,555 Sri Lankan catalog products from 20 stores
- 4,324 indexed apparel documents after 231 non-apparel products were excluded
- Shopify and WooCommerce snapshots ingested through Samesake connectors
- Woo minor-unit price normalization
- 50 golden fashion-search queries
- the same ESCI LLM judge used by the spike harness

The enrichment run live-processed 1,327 products through the Samesake pipeline, then imported the remaining enrichment and embedding rows from the spike tables for cost control. That means the benchmark proves retrieval parity on the shared indexed catalog; it does not prove that every enrichment row was freshly regenerated by Samesake in that run.

## Method

The comparison used the external aggregator harness:

```bash
cd ../project-search-web-search
node scripts/eval-search.js --target v2 --base http://localhost:8788
```

The result files recorded in the aggregator repo were:

- `evals/results/v2-spike-baseline.json`
- `evals/results/v2-samesake-2026-06-11.json`
- `evals/results/fanout-2026-06-10.json`

## Results

| Metric | Gate | Samesake | Spike v2 | Fan-out v1 |
|---|---:|---:|---:|---:|
| mean grade@10 | >= 2.30 | 2.338 | 2.422 | 1.470 |
| nDCG@10 | - | 0.935 | 0.941 | 0.820 |
| P@5, grade >= 2 | >= 0.78 | 0.804 | 0.828 | 0.480 |
| price-violation rate | <= 0.20 | 0.16 | 0.16 | 0.62 |
| zero-result rate | - | 0.00 | 0.00 | 0.00 |
| median latency | - | 1,065 ms | 1,339 ms | 8,784 ms |

The acceptance gate was met. Samesake reproduced the spike's quality within tolerance and beat the original fan-out v1 baseline by a wide margin on every listed metric.

## What The Benchmark Shows

Keyword-only retrieval is not enough for fashion because shopper language is vague and often describes use, style, material, or occasion rather than product names.

Vector-only retrieval is not enough because commerce search needs hard constraints. A visually or semantically close result that violates price, availability, gender/category, or store policy is still a bad result.

Flat visual similarity is not enough because "looks similar" is only one signal. Fashion search also needs size, color, material, occasion, price, freshness, availability, and merchandising policy.

Samesake's measured path combines:

- FTS for literal product and attribute matches
- text embeddings for intent residuals
- hard SQL filters for constraints
- facets and pagination for commerce UX
- enrichment for normalized shopper-facing attributes
- RRF ranking so multiple retrieval legs can contribute without one score scale dominating

## Failure Modes

The parity work found concrete framework issues and fixed them:

- `alsoMatch` enum filters initially rendered duplicate SQL parameter indexes, causing Postgres `42P18` on gender-bearing queries.
- Example execution depended on built package `dist/`, so source fixes needed a package build before examples reflected them.
- Local/corpus-coverage queries remained a weak spot for both the spike and Samesake.

## What This Does Not Prove Yet

This benchmark does not prove:

- mature reverse-image search quality across arbitrary inspiration images
- personalization or user taste modeling
- performance at tens of millions of products
- production dominance over tuned Algolia, Elastic, or custom marketplace systems
- merchant-specific ranking quality without merchant-specific tuning
- that all enrichment can be run cheaply at full catalog scale

Those are separate proof points and should stay out of public claims until measured.
